# 心理健康 App 项目上下文

## 文档目的

本文件用于统一当前项目的产品定位、MVP 范围、技术路线、数据边界与开发优先级，作为后续需求拆解、架构设计、接口设计和代码实现的基线。

信息来源：

- `PLAN.md`
- `plan.png`

## 当前统一结论

### 1. 产品定位

项目定位为“心理健康筛查辅助 + AI 康复训练陪伴 + 家校摘要预警”的试点型 MVP。

明确边界：

- 不做医学诊断。
- 不让摄像头单独判断用户是否存在心理疾病。
- 高风险/危机内容必须进入人工复核流程。
- 优先做本地/演示试点，不直接面向正式生产环境。

### 2. 阶段目标用户

第一阶段统一按“中国大陆中学未成年人试点”推进。

补充说明：

- `plan.png` 中提到“中学生和大学生”。
- `PLAN.md` 对未成年人场景、监护授权、学校心理老师参与、数据最小化要求定义更完整。
- 因此当前以“中学生未成年人”作为 MVP 主场景，大学生场景放入后续扩展范围。

### 3. 角色与端

项目至少包含三类使用端：

- 学生端 App
- 家长端
- 老师/学校心理老师端

后台还需支持管理角色：

- `admin`
- `school_counselor`
- `teacher`
- `guardian`
- `student`

### 4. 核心业务闭环

MVP 的核心闭环应为：

1. 学校导入学生、班级、老师信息。
2. 生成学生账号与家长绑定邀请码。
3. 完成监护人同意、学生知情同意及专项授权。
4. 学生完成基础筛查/量表/自评。
5. 系统输出风险趋势与训练建议。
6. 学生进入 AI 陪练或结构化训练任务。
7. 家长端查看摘要与提醒。
8. 老师端接收高风险预警并进行人工复核。

## MVP 范围

### 学生端

- 登录与身份绑定
- 知情同意与授权管理
- 基础筛查/量表提交
- AI 陪练对话
- 训练任务执行与打卡
- 情绪记录/简要自评
- 训练结果与趋势查看

### 家长端

- 绑定学生
- 查看摘要信息
- 查看趋势提醒
- 接收异常提醒

限制：

- 不展示完整聊天记录
- 不展示原始视频
- 不展示高隐私日记全文

### 老师/学校心理老师端

- 查看班级或个体摘要
- 查看高风险预警单
- 标记复核状态
- 查看必要上下文、风险原因、建议动作

限制：

- 不能默认查看完整私密内容
- 只能按最小必要原则访问数据

## 风险识别与 AI 原则

### 风险识别

风险判断以以下信息为主：

- 量表/问卷
- 学生文本自评
- 情绪趋势与训练完成情况
- 规则命中
- 模型辅助判断
- 必要的人工复核

明确排除：

- 摄像头结果不直接参与心理风险分级
- 视频数字人信号不作为心理诊断依据

### AI 陪练

AI 能力定位：

- 生成训练话术
- 提供鼓励反馈
- 辅助生成训练计划
- 做安全改写与风险拦截

第一阶段原则：

- App 不直连大模型
- 后端统一提供 AI 安全网关
- 第一阶段先接 Mock/沙箱
- 后续通过适配层接入真实供应商

### 数字人

数字人能力仅作为训练交互体验增强，不作为风险判断核心。

MVP 原则：

- 可先使用平台默认模板或授权头像
- 所有生成内容需要显著标注“AI 生成/数字人生成”
- 不允许未经授权上传他人肖像生成误导性数字人

## 技术路线

### 后端

统一以 `Java 17 + Spring Boot 3.x` 先落地，后续可平滑升级 Java 21，配套：

- MySQL 8
- H2 本地开发数据库
- Redis
- Flyway
- Spring Security
- Docker Compose

建议优先搭建后端骨架，因为该项目的权限、授权、审计、风险流转都强依赖后端模型设计。

后端核心模块：

- 学校租户
- 学生/家长/老师账号
- 邀请码绑定
- 监护授权与撤回
- 筛查记录
- 风险分级引擎
- 训练计划与训练记录
- AI 网关
- 数字人网关
- 预警单与人工复核
- 审计日志

### 前端

移动端建议使用 Flutter，当前方案为：

- Flutter
- Riverpod
- GoRouter
- Dio
- secure_storage

界面按学生、家长、老师三类入口拆分。

## 数据与合规边界

### 数据最小化

默认遵守以下原则：

- 不保存原始视频
- 默认只保存必要摘要、风险命中、训练建议与审计事件
- 摄像头尽量本地侧处理
- 所有查看行为写入审计日志

### 授权机制

需要记录的授权类型至少包括：

- 监护人同意
- 学生知情同意
- 摄像头训练授权
- 个人形象用于数字人的单独授权

授权记录应保存：

- 版本号
- 时间
- 来源
- 当前状态
- 撤回状态

### 高风险处理

当出现自伤、自杀、被伤害等高风险表达时：

- 自动创建预警单
- 触发人工介入流程
- 提示联系可信成年人/学校心理老师/紧急服务

### 单元测试

- 风险分级
- 危机词命中
- 摄像头不参与风险分级
- 权限隔离
- 授权撤回
- 训练计划生成

### 集成测试

- 学校导入
- 邀请码绑定
- 三端登录
- 筛查提交
- AI Mock
- 数字人 Mock
- 预警单创建与处理

### 安全测试

- 越权访问
- 未授权读取
- 接口限流
- 审计日志完整性
- 删除/撤回授权后的访问拦截

## 当前仍需确认的问题

以下问题尚未完全定稿，后续需要在正式开发前确认：

- 学生端是否只覆盖中学生，还是保留大学生模式开关
- 风险分级具体采用哪些量表与规则库
- 学校导入方式是 Excel、CSV 还是管理后台手工录入
- 家长端与老师端是独立 App、H5 还是同一 Flutter 多角色应用
- 数字人第一阶段是纯 Mock、TTS 虚拟形象，还是接供应商沙箱
- 是否需要离线部署或校内部署能力

## 建议的开发优先级

建议按以下顺序推进：

1. 后端领域模型与权限模型
2. 授权与账号绑定流程
3. 筛查、风险分级与预警单
4. AI 网关与 Mock 适配
5. Flutter 三端基础壳与核心流程
6. 数字人能力接入
7. 审计、删除与导出机制
