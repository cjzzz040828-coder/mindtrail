# 心理健康筛查辅助与康复训练 App 方案图对齐说明

## 来源

本说明基于用户在 2026-04-24 提供的方案图整理，目的是把图中的产品边界、技术思路和测试重点，转成当前仓库可直接采用的统一描述。

## 图中核心结论

### 1. 产品定位

- 产品定位为“心理健康筛查辅助 + AI 训练陪伴 + 家校摘要预警”。
- 明确不做医学诊断。
- 不允许摄像头单独判断用户是否存在心理疾病。
- 家长端和老师端采用摘要模式，不展示完整聊天、原始视频和高隐私日记全文。

### 2. 目标用户

- 图中目标用户包含中学生和大学生。
- 结合当前项目实现和合规要求，MVP 仍以中国大陆中学未成年人试点为主。
- 中学生场景必须具备监护人授权、学生知情同意、学校心理老师或辅导员参与、最小化数据共享。
- 大学生模式可以作为后续扩展，而不是当前第一优先级。

### 3. 风险与识别原则

- 筛查应以成熟量表、文本自评、睡眠/压力记录、训练完成情况和必要人工复核为主。
- 摄像头识别仅作为训练过程中的辅助信号，例如是否完成大笑、呼吸、放松训练，或是否存在注意力波动、表情紧绷、状态疲劳等。
- 摄像头和数字人信号不参与心理风险分级，不输出“抑郁症、焦虑症”等诊断标签。
- 当用户出现自伤、自杀、被伤害等高风险表达时，系统应立即给出求助提醒，并触发人工介入流程。

### 4. 康复训练方向

- AI 负责生成结构化训练计划，例如：
  - 大笑训练
  - 正念呼吸
  - 情绪日记
  - 认知重构
  - 行为激活
  - 睡眠改善
  - 轻运动/拉伸
- 数字人只用于训练引导和互动体验增强，不作为风险判断主体。
- 行为建议可以扩展到音乐建议、膳食建议、微任务建议，但都应使用“辅助建议”口径，避免被描述成医疗干预。

### 5. 数据与授权

- 默认不上传原始视频，只上传训练结果或必要特征。
- 摄像头应只在用户主动发起训练时开启。
- 数字人头像必须使用平台默认模板或用户本人明确授权的形象。
- 禁止上传他人照片用于误导性数字人生成。
- 所有查看行为、状态更新和关键判断流程都应有审计日志。

## 与当前项目的对齐情况

### 已落地

- 学生端：登录、授权、筛查、训练计划、AI 陪练。
- 家长端：摘要、趋势、提醒、隐私边界展示。
- 老师端：预警列表、预警详情、状态更新。
- 后端：授权、筛查、训练、AI 陪练、状态观察建议、审计日志、MySQL/H2/Flyway 持久化。
- 安全边界：不做医学诊断、不保存原始视频、高风险进入人工复核、家长和老师只看必要摘要。

### 已按图中原则实现

- `AI 陪练` 已通过后端安全网关返回非诊断性回复。
- `状态观察建议` 已实现“表情/行为指标 -> 可能状态 -> 行为建议/音乐/膳食建议 -> 数字人引导话术”的安全版闭环。
- `状态观察建议` 明确声明：
  - 仅用于训练辅助
  - 不构成医学诊断
  - 不参与风险分级
  - 不保存原始图片、视频或完整私密内容

### 仍需继续补强

- 学校导入、邀请码绑定、角色权限的完整链路。
- 监护人撤回授权后的访问拦截和数据可见性变化。
- 数字人训练页的专门入口、头像授权校验和水印提示。
- 音乐建议、膳食建议的精细规则库和可配置后台。
- 更完整的测试矩阵，包括不同光照、角度、遮挡场景下的状态观察容错。

## 当前推荐表述

为了降低误判和合规风险，后续在页面、接口和对外介绍中，建议统一使用下面这些表述：

- 使用“筛查辅助”而不是“心理诊断”。
- 使用“状态观察建议”而不是“表情识别判断心理疾病”。
- 使用“可能存在注意力疲劳 / 紧张状态 / 心理能量偏低”而不是“已经抑郁 / 焦虑 / 消极厌世”。
- 使用“训练建议、音乐建议、膳食建议”而不是“治疗方案”。
- 使用“高风险内容进入人工复核”而不是“系统自动判定结论”。

## 下一步建议

按这张方案图和当前代码状态，后续最值得继续做的是：

1. 增加“数字人训练模式”页面，把状态观察建议、AI 陪练和训练动作整合成一个完整训练回合。
2. 增加“学校导入 + 家长绑定 + 角色权限”闭环，让三端演示更完整。
3. 增加“音乐建议/膳食建议”的可配置规则，便于后续快速调优展示效果。
4. 增加“授权撤回、越权访问、危机求助”的专项测试用例，强化试点演示可信度。
