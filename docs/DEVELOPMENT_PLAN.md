# 心理健康 App MVP 开发计划

## 计划目标

本计划用于把当前原型图与已有工程骨架推进成一个可演示、可联调、可继续扩展的 MVP。

当前目标不是一次性做成正式上线产品，而是先完成一条可信的试点闭环：

```text
学校/学生导入 -> 学生登录 -> 授权同意 -> 筛查自评 -> 风险趋势 -> 训练计划 -> 家长摘要 -> 老师预警复核
```

## 当前基线

已经完成：

- 产品方案与边界文档：`PLAN.md`
- 项目上下文文档：`PROJECT_CONTEXT.md`
- 原型图素材：`pictures/`
- 后端 Spring Boot 骨架：`backend/`
- Vue 3 H5/Web 前端：`frontend_vue/`
- Flutter 入口与学生端最小流程骨架：`app_flutter/`
- 后端 API 与第一版持久化：
  - 学生登录
  - 授权状态
  - 筛查模板
  - 筛查提交
  - 今日训练
  - 训练任务完成状态更新
  - 家长摘要
  - AI 陪练安全网关 Mock
  - 表情/行为状态观察建议 Mock
  - 老师预警
  - 老师预警状态更新
  - H2 默认数据库
  - MySQL 8 联调/部署 profile
  - Flyway 核心表迁移
  - 基础审计日志记录
  - 学生训练完成率联动家长摘要
  - AI 危机表达触发人工复核预警
  - 状态观察仅用于训练建议，不参与风险分级

当前限制：

- Vue 前端已可构建，Flutter SDK 尚未在本机验证。
- 后端已经接入数据库，并完成家长摘要、老师预警状态更新和基础审计日志；账号权限、处置详情仍需继续补全。
- 家长端、老师端还只是入口骨架。
- AI 陪练已有后端安全网关 Mock，状态观察建议已有安全版 Mock，数字人还未接真实供应商。

## 开发原则

- 先做可跑通的核心闭环，再做视觉精修。
- 所有涉及未成年人隐私的能力都默认最小化展示。
- 摄像头和数字人能力只服务训练体验，不参与心理风险诊断。
- 风险判断先规则化、可解释化，后续再引入模型辅助。
- App 不直连模型，所有 AI 能力都走后端网关。
- 高风险表达必须生成预警单，并进入人工复核。

## 阶段一：学生端闭环可运行

预计周期：1 周

目标：

- 让学生端从登录到训练首页可以在 Vue 前端中实际跑通。
- 后端接口保持 Mock，但返回结构稳定。

主要任务：

- 跑通 `frontend_vue` 的 `npm install`、`npm run dev`、`npm run build`。
- 修正 Vue 页面联调问题和跨域问题。
- 优化学生登录页、授权页、筛查页、结果页、训练页。
- 给学生端关键页面增加加载、错误、空状态。
- 后端补接口集成测试。

验收标准：

- 后端 `mvn test` 通过。
- App 能进入学生端。
- App 能完成登录、授权、筛查、查看结果、进入训练计划。
- Android 模拟器或桌面端至少一种方式能跑通。
  如果优先 H5，则浏览器能完整跑通即可。

## 阶段二：数据库与真实数据模型

预计周期：1 到 2 周

目标：

- 把核心 Mock 数据迁移到 MySQL 8 可部署的数据模型。
- 建立能支撑试点的基础数据模型。

主要任务：

- 引入 MySQL 8、Flyway、Spring Data JPA。
- 保留 H2 默认开发数据库，便于无数据库环境演示。
- 增加 Docker Compose 或本地 MySQL 启动说明。
- 设计并创建核心表：
  - `schools`
  - `classes`
  - `users`
  - `student_profiles`
  - `guardian_bindings`
  - `consent_records`
  - `screening_templates`
  - `screening_submissions`
  - `risk_assessments`
  - `training_plans`
  - `training_tasks`
  - `alert_cases`
  - `audit_logs`
- 将学生登录、授权、筛查、训练计划改为读写数据库。
- 加入基础审计日志。

验收标准：

- 一条 Docker 命令可启动数据库。
- Flyway 可自动建表。
- 学生端提交的数据能在数据库中查到。
- 授权撤回后，相关接口能体现状态变化。

## 阶段三：家长端 MVP

预计周期：1 周

目标：

- 家长能绑定学生并查看摘要。
- 家长端只展示趋势与建议，不展示完整聊天、隐私日记、原始视频。

主要任务：

- 后端实现家长邀请码绑定。
- 后端实现家长摘要接口。
- Flutter 实现家长端首页摘要。
- Flutter 实现趋势详情、提醒事项、授权与沟通页。
- 增加家长端权限校验。

验收标准：

- 家长能通过邀请码绑定学生。
- 家长能看到训练完成率、情绪趋势、提醒事项。
- 家长不能读取学生完整私密内容。

## 阶段四：老师端与预警复核

预计周期：1 到 2 周

目标：

- 老师或学校心理老师能查看预警、处理预警、记录处置过程。

主要任务：

- 实现风险预警单表与状态流转。
- 实现老师端预警列表。
- 实现预警详情与复核动作。
- 实现处置记录。
- 实现审计筛选。
- 增加角色权限：
  - 普通老师只能看授权范围内摘要。
  - 学校心理老师可看必要上下文。
  - 管理员可做学校级配置。

验收标准：

- 高风险筛查结果能创建预警单。
- 老师能标记处理中、已复核、已归档。
- 预警处理动作会写入审计日志。
- 老师端不展示完整聊天和原始视频。

## 阶段五：AI 陪练与安全网关

预计周期：1 到 2 周

目标：

- 建立 AI 陪练接口和安全边界，第一版使用后端 Mock 网关。

主要任务：

- 继续完善 AI 会话接口：
  - 创建会话
  - 发送消息
  - 返回陪练回复
  - 返回风险命中
  - 返回训练建议
- 增加安全规则：
  - 危机词命中
  - 自伤自杀表达拦截
  - 医疗诊断话术限制
  - 转人工建议
- Flutter 实现 AI 陪练聊天页。
- 默认只保存摘要、风险命中、训练建议和审计事件。

验收标准：

- AI 陪练页面可用 Mock 回复跑通。
- 危机表达能触发预警。
- 回复中不出现医学诊断承诺。
- 后端保留可替换真实模型供应商的接口层。

## 阶段六：数字人与训练体验

预计周期：1 周

目标：

- 数字人只作为训练陪伴体验，不作为风险判断来源。

主要任务：

- 设计数字人网关接口。
- 增加数字人授权校验。
- Flutter 实现数字人陪练入口。
- 所有数字人内容显著标记 AI 生成。
- 暂用 Mock 或静态头像，不直接接生产供应商。

验收标准：

- 未授权时无法进入数字人陪练。
- 授权后可以进入 Mock 数字人训练页。
- 页面显著展示 AI 生成提示。

## 阶段七：演示打磨与试点准备

预计周期：1 周

目标：

- 准备一版可给学校、家长或合作方演示的 MVP。

主要任务：

- 按原型图统一视觉风格。
- 增加演示数据脚本。
- 增加接口文档。
- 增加部署说明。
- 补充隐私说明、免责声明、授权文案。
- 准备演示流程：
  - 学生完成筛查
  - 生成训练计划
  - 家长查看摘要
  - 老师处理预警

验收标准：

- 10 分钟内能完整演示一条学生案例。
- 演示过程中不需要手动改数据库。
- 核心隐私边界能在页面和文档中解释清楚。

## 第一周详细任务

优先级 P0：

- 启动 Spring Boot 后端。
- 跑通 Vue 前端开发服务器。
- 用浏览器完成学生端完整流程。
- 后端补学生端接口集成测试。
- 学生端登录、授权、筛查、结果、训练页面可实际跳转。

优先级 P1：

- 页面视觉向原型图靠拢。
- 增加统一颜色、间距、卡片组件。
- 增加错误重试与加载状态。
- README 增加更细的本地启动说明。

优先级 P2：

- 增加家长端占位接口。
- 增加老师端占位接口。
- 继续完善数据库表设计和接口测试。

## 主要风险

- 合规风险：未成年人数据、监护授权、人脸/头像授权都需要谨慎处理。
- 产品风险：容易被误解为医学诊断工具，需要持续强调筛查辅助与人工复核。
- 技术风险：AI 与数字人供应商 API、费用、备案、数据出境要求未确定。
- 试点风险：学校真实流程中可能需要审批、家长通知和心理老师工作机制配合。

## 暂不做

第一版 MVP 暂不做：

- 正式应用商店上架
- 真实医学诊断
- 保存原始视频
- 学生任意上传他人肖像
- 真实生产 AI 或数字人供应商接入
- 大规模学校多租户运营后台

## 下一步执行建议

从现在开始，最好的下一步是：

```text
先让 Vue 前端和 Spring Boot 后端一起跑起来，然后把学生端闭环调通。
```

如果后续要正式做原生 App，再决定使用 `uni-app`、`Capacitor` 或继续推进 Flutter。
