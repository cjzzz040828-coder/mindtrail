# 心理健康 App 试点 MVP 技术方案

## 摘要

- 正式产品从零搭建：当前 `React/Vite` 项目只作为原型图和产品蓝图参考。
- 第一阶段目标：面向中国大陆中学未成年人，做“学生 App + 家长端 + 老师端”的本地/演示试点 MVP。
- 技术栈：Flutter App + Java/Spring 后端，优先搭后端骨架。
- AI 方案：App 不直连模型；后端提供大模型安全网关和视频数字人供应商适配层，第一阶段先做 Mock/沙箱，后续替换真实 API。
- 安全边界：不做医学诊断，不保存原始视频，最小必要保存；高风险/危机表达触发学校心理老师人工复核。

## 关键实现

- 创建项目上下文文档 `D:\desktop\心理健康\PROJECT_CONTEXT.md`，记录本轮已确认决策、技术栈、数据边界和后续开发入口。
- 后端当前采用 `Java 17 + Spring Boot 3.x + MySQL 8 + Redis + Flyway + Spring Security` 路线，默认本地开发可用 H2 免安装数据库，后续可用 Docker Compose 启动数据库、缓存和服务。
- 后端模块包括：学校租户、学生/家长/老师账号、邀请码绑定、监护授权、筛查风险引擎、训练记录、AI 陪练网关、视频数字人网关、预警复核、审计日志。
- Flutter 采用 `Riverpod + GoRouter + Dio + secure_storage`，按学生、家长、老师三类入口拆分页面。
- 少量真实学生数据只能在完成学校授权、监护人同意、学生知情同意、删除机制和审计留痕后进入演示环境。

## 接口与数据

- 账号接口：学校导入学生/班级/老师，生成学生账号和家长绑定邀请码；角色为 `student`、`guardian`、`teacher`、`school_counselor`、`admin`。
- 授权接口：记录监护人同意、学生知情同意、摄像头训练授权、本人形象用于数字人的单独授权；每次授权保存版本号、时间、来源和撤回状态。
- 筛查接口：提交自评量表和文本备注，返回低/中/高风险趋势、原因、免责声明、下一步建议；摄像头/视频数字人信号不参与心理风险分级。
- AI 陪练接口：创建会话、发送消息、返回安全改写后的陪练回复；默认不长期保存完整聊天，只保存必要摘要、风险命中、训练建议和审计事件。
- 数字人接口：创建视频数字人会话，使用学生本人授权形象或供应商授权模板；所有生成内容必须带“AI 生成/数字人生成”显著提示。
- 复核接口：高风险/危机表达自动创建预警单，学校心理老师只能看摘要、风险原因、建议动作和必要上下文，不能看完整隐私日记或原始视频。

## 测试与验收

- 单元测试：风险分级、危机词命中、摄像头不参与风险分级、权限隔离、授权撤回、训练计划生成。
- 集成测试：学校导入、邀请码绑定、三端登录、筛查提交、AI 网关 Mock、数字人 Mock、预警单创建与处理。
- 安全测试：越权访问、未授权数据读取、接口限流、审计日志完整性、删除/撤回授权后的访问拦截。
- App 验收场景：学生完成授权、筛查、AI 陪练、训练打卡；家长查看摘要；老师接收高风险预警并标记复核状态。
- 本地验收标准：一条 Docker 命令启动后端依赖，Spring 测试通过，Flutter 可跑通三端核心流程，Mock AI/数字人可替换为真实供应商适配器。

## 假设与依据

- 默认第一阶段不做正式上架、不接入真实生产供应商、不保存原始视频、不开放学生任意上传他人肖像。
- 大模型和视频数字人供应商暂不锁死，通过后端接口适配；真实接入前再确定 API、费用、备案、数据出境和内容安全责任。
